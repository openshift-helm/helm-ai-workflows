HELM-123: Add timeout configuration support for Helm chart installations

Description:
Currently, Helm chart installations use a hardcoded timeout value. Users need the ability to
specify custom timeout values for long-running chart installations.

Problem:
- Some charts with complex initialization scripts take longer than the default timeout
- Users cannot customize the timeout value per installation
- This causes installation failures for legitimate long-running deployments

Steps to Reproduce:
1. Install a Helm chart that takes more than 5 minutes to become ready
2. Observe that the installation times out and fails
3. No option exists to configure a longer timeout

Actual Results:
Installations timeout after the default 5 minutes with no way to configure this.

Expected Results:
Users should be able to specify a custom timeout value when installing charts.

Acceptance Criteria:
1. Add a `timeout` field to the HelmRequest structure
2. Update HandleHelmInstall to accept and use the custom timeout value
3. If no timeout is specified, use the existing default (300 seconds)
4. Validate that timeout values are within reasonable bounds (30s - 1800s)
5. Add tests to verify custom timeout handling

Technical Details:
- Modify the HelmRequest struct in pkg/helm/handlers/request.go
- Update installation handler to pass timeout to Helm client
- Add validation for timeout range
- Update API documentation
